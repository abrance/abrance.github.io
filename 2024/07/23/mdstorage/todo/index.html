HA 授权优化

需求资料

3）引擎工作节点（SE-W）
流量引擎的工作节点由于存在单独购买以进行节点扩容的情况，基于当下的授权机制，存在时间授权的情况。
流量引擎工作节点的过期判定逻辑为：
●授权有效期结束。
过期前的处理机制：
●当剩余有效期低于90天、60天、30天、15天时，触发通知机制：
○发送邮件通知，内容：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxxx）的授权将于YYYY-MM-DD过期，请及时续费，到期未续费将会影响您的使用。
■其中：aaaaa为引擎节点名称
■xxxx为引擎工作节点的序列号
■YYYY-MM-DD为该引擎工作节点的授权过期时间，取自授权许可文件
■基于通知策略配置发送，若未配置则不发送
■若存在多个引擎工作节点的有效期一致，将多个引擎聚合到一条通知中发送，样式为：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxx）、『bbbbb』（序列号：xxxxx）、『ccccc』（序列号：xxxxxx）的授权将于YYYY-MM-DD过期……（不同序列号为不同引擎工作节点）
○发送站内信通知，内容：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxxx）的授权将于YYYY-MM-DD过期，请及时续费，到期未续费将会影响您的使用。
■存在多个引擎工作节点的有效期一致时，将多个引擎聚合到一条通知中发送（参考邮件的举例）
○在触发时间点，自动执行相应的通知机制：
■若执行失败则重试
■若重试失败，则第二天再度触发，直到：执行成功或进入第二个触发点
■每个触发点范围内，只发送一次，发送成功后则到下一个触发点才会再次发送
■邮件和站内信分开处理，若1个成功1个失败，则成功的不需要重试，失败的需要重试
●当剩余有效期＞1天且＜15天时：
○系统管理员进入系统时，在页面顶部显示常驻异常toast，文案为：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxxx）的授权将于YYYY-MM-DD过期，请及时续费，到期未续费将会影响您的使用。
○该toast遵循常驻toast的交互逻辑，具体见通用交互规范；
○该toast每天仅显示1次，当被关闭后或退出登录后当天不再显示；
○该toast切换页面时保持常驻不消失；
○当授权更新后，剩余有效期不在此范围内了，则不进行相应的提示。
●当剩余有效期=1天时：
○发送邮件通知，内容：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxxx）的授权将于YYYY-MM-DD过期，请及时续费，到期未续费将会影响您的使用。
○发送站内信，内容：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxxx）的授权将于YYYY-MM-DD过期，请及时先续费，到期未续费将会影响您的使用。
○管理员账号每次登录系统时，弹框提示，文案为：您购买的流量引擎工作节点『aaaaa』（序列号xxxxx）、『bbbbb』（序列号：xxxxx）的授权将于YYYY-MM-DD过期，请及时续费，到期未续费将会影响您的使用。点击「知道了」按钮，关闭弹框；点击「去看看」，跳转页面『系统管理-授权信息』，并展开所在工作节点的弹框。

过期时的处理机制：
●发送邮件通知，内容：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxx）的授权已过期，为避免影响您的正常使用，请及时续费。
●发送站内信，内容：您购买的流量引擎工作节点『aaaaa』（序列号：xxxxxx）的授权已过期，为避免影响您的正常使用，请及时续费。
管理员账号登录系统成功时，弹框提示，文案为：您购买的流量引擎工作节点『aaaaa』（序列号xxxxx）、『bbbbb』（序列号：xxxx）的授权已过期，为避免影响您的正常使用，请及时续费。点击「知道了」按钮，关闭弹框；点击「去看看」，跳转页面『系统管理-授权信息』，并展开所在工作节点的弹框。

○提供勾选项：不再提示。默认不勾选
■不勾选时，管理员账号每天首次登录系统时都将进行弹框提示（每天一次）
■勾选后，授权更新前不再进行提示
○过期时若管理员账号在线，则立即通知；若管理员账号不在线，则过期后首次登录登录时执行第一次提示。
●授权过期的引擎工作节点：
○识别到过期后，引擎需调度不再向该节点分发流量，已分发的流量需继续处理完成；
○已分发的流量处理完成后，该节点在授权续期前将停止工作，运行状态标记为“故障”，故障原因为“工作节点授权过期”。
○过期的工作节点可以进行正常的管理，包括删除、替换、配置下发等，但其不生效。
○已过期的引擎工作节点不能安装，相关按钮禁用并不计入可安装工作节点。
授权状态标记：
●剩余有效期大于30天时，状态标记为：授权正常
●剩余有效期小于30天时，状态标记为：即将过期
●过期后，状态标记为：授权过期
其他补充：
●若引擎中只有一个工作节点，则该节点在过期时也不能停止工作（即必须至少保证引擎内有一个可以正常工作的工作节点），需保持其正常工作，直至新的工作节点加入后，该节点才能遵循过期时的机制，停止工作；此时，其状态保持不变，不需要标记为故障。
●若所有工作节点同时过期，随机选择一个保留。

整理的机制

  • 每个 worker 需要知道自己的授权状态，上层帮忙维护， : auth_status: ok | near_expired | expired
  • 每个 worker 需要根据此值维护自己的工作状态 ：work_status: ok | nok (维持过期后即不工作，但是在所有工作节点过期后，至少有一台工作节点工作)
  • 每个 worker 需要维护自己的工作状态是否异常，如果异常则可能需要上层管理应用控制其他 worker 工作 : status: ok | nok

总控 IP 修改

概述

场景描述： 总控机器的网络环境需要发生变化，如机器有 192.168.0.1 -> 172.24.0.1 网络路由已经发生了变化，此时总控引擎失联。需要在通知到引擎，总控 ip 修改的信息，使总控引擎重新建立连接。

实现步骤

  • 保证总控机器内所有应用对机器 ip 修改不敏感（如果敏感将导致总控修改 ip 后应用不正常或挂掉的风险），或者能适配允许的任何 ip 方面的修改（这个目前不熟悉这块）
  • 引擎中的应用能接收总控 ip 修改的信息
  • 引擎所有节点的所有应用都采用新的 ip 与总控的应用重新建立连接

信创（飞腾）

鲲鹏920是目前业界领先的ARM-based处理器。该处理器采用7nm制造工艺，基于ARM架构授权，由华为公司自主设计完成。通过优化分支预测算法、提升运算单元数量、改进内存子系统架构等一系列微架构设计，大幅提高处理器性能。典型主频下，SPECint Benchmark评分超过930，超出业界标杆25%。同时，能效比优于业界标杆30%。鲲鹏920以更低功耗为数据中心提供更强性能¹。

飞腾S2500是目前国产性能最强的多路服务器系统。作为16nm工艺CPU，性能已跟7nm的华为鲲鹏920难分伯仲。而CPU制作工艺缩小会提高CPU性能，预计明后年发布的7nmS5000、5nmS6000将会在性能上有质的飞跃³.

两者之间的性能差异值得深入思考。例如，鲲鹏920拥有64个核心，而飞腾S2500则有16个NUMA节点。较少的NUMA节点意味着访存一致性更好。此外，鲲鹏的缓存量相对较少，尽管一般来说，缓存越多性能越好，但怀疑飞腾的多核之间的通信可能不够优化，导致缓存总数较多⁴。

总之，这两款处理器在不同方面都有各自的优势，具体应用场景和需求会影响选择。⁵

mmap 机制

mmap 通过将文件直接映射到进程的地址空间来提高文件访问的效率。

优势

  1. 性能：对于大文件的随机读写，mmap 可以提供更高的性能，因为它避免了传统的文件读写中的数据复制步骤。
  2. 简易的接口：mmap 提供了一种将文件内容直接映射为内存的方法，这使得读写文件就像访问内存一样简单。
  3. 系统优化：利用操作系统的页面缓存和虚拟内存系统，可以有效地处理大文件。

劣势

  1. 安全风险：直接操作内存可能导致程序崩溃和数据损坏。
  2. 复杂性：需要手动管理内存映射的生命周期，错误处理更加复杂。
  3. 可移植性和兼容性问题：不同操作系统对 mmap 的支持不同，可能需要平台特定的代码。（检测适配 鲲鹏 kylinos v10）

适用场景

  • 对大型文件进行高效的随机读写。
  • 处理有限的结构化数据，不需要复杂的查询或事务处理。
  • 在内存和文件之间快速共享数据。

数据库（DB）

数据库是用于存储、检索、更新和管理数据的系统，它提供了数据的结构化组织、查询语言和事务处理。

优势

  1. 数据管理：提供复杂的数据查询、索引和管理能力。
  2. 事务支持：支持事务处理，包括原子性、一致性、隔离性和持久性。
  3. 易于使用和维护：提供了用户友好的接口和工具，简化了数据管理和维护。
  4. 扩展性和灵活性：支持大量数据和多种数据类型，易于扩展和修改数据模型。
  5. 安全性：提供访问控制和数据加密功能。

劣势

  1. 性能开销：对于简单的数据访问，数据库的额外功能可能带来性能开销。
  2. 复杂性：数据库的安装、配置和维护相对复杂。
  3. 成本：某些数据库软件可能需要购买许可证。

适用场景

  • 需要复杂查询和数据分析的应用。
  • 需要事务处理和数据完整性保证的场景。
  • 处理大量结构化数据和关系型数据。
  • 需要多用户并发访问和数据安全性的应用。

epoll 调用

创建销毁生命周期

Linux 中的 epoll 是一种高效的 I/O 事件通知机制，广泛用于处理大量并发网络连接。它的生命周期主要包含以下几个阶段：

  1. 创建 epoll 实例：使用 epoll_create 或 epoll_create1 函数创建一个新的 epoll 实例。这将返回一个文件描述符，用于所有后续的 epoll 操作。

  2. 添加文件描述符：通过 epoll_ctl 函数，你可以向 epoll 实例添加文件描述符，并指定你关心的事件（如读就绪、写就绪等）。这个步骤可以重复执行，以监控多个文件描述符。

  3. 等待事件：调用 epoll_wait 或 epoll_pwait 函数等待事件的发生。这些函数会阻塞直到至少有一个监控的文件描述符上发生了指定的事件，或者达到了超时时间。

  4. 处理事件：epoll_wait 返回时，它会提供一个事件列表，包含自上次调用以来发生的所有事件。应用程序需要遍历这个列表，并对每个事件进行处理。

  5. 修改文件描述符：如果需要更改对某个文件描述符的监控事件或从监控列表中移除文件描述符，可以再次使用 epoll_ctl 函数。

  6. 关闭 epoll 实例：使用完 epoll 实例后，应该通过关闭其文件描述符来释放资源，通常是调用 close 函数。

架构演进

技术和业务之间如何影响

产品类业务

  1. 定义和特点：产品类业务指的是那些提供具体、独立的产品给消费者的业务，如杀毒软件、智能手机、浏览器等。这些产品具有明确的功能，用户可以通过购买或免费使用它们来完成特定的任务。

  2. 用户关系：用户与这些产品的关系是“独占”的。即一旦购买或下载，用户对该产品有完全的使用权，不受其他用户的使用影响。

  3. 商业模式：这类产品通常依靠销售收入或通过其他方式（如广告、内购）获利。它们与传统制造业产品在商业模式上有一定的相似性。

  4. 市场策略：产品质量、创新、品牌和营销策略对于成功至关重要。产品类业务需要不断迭代更新，以适应市场的变化和用户的需求。

服务类业务

  1. 定义和特点：服务类业务提供的是可以由多个用户同时使用的服务，如搜索引擎、在线购物、社交网络等。这些服务依赖于用户之间的互动和交流。

  2. 用户关系：用户“使用”而不“独占”这些服务。一个用户的使用不会阻碍其他用户同时使用该服务。

  3. 商业模式：服务类业务的价值随着用户数量的增加而增加，符合网络效应原理。这种业务模式非常依赖于用户基数，通常通过广告、交易费用或订阅模式盈利。

  4. 市场策略：对于服务类业务而言，用户参与度、用户体验和社区建设至关重要。此外，数据的收集和分析对于提供个性化服务和改善用户体验非常重要。

综合分析

  • 业务本质：产品类业务更侧重于提供功能性的实体或数字产品，而服务类业务侧重于创建一个平台或环境，让用户在其中进行交互和活动。

  • 增值方式：产品类业务的增值通常来自于产品本身的改进和创新，而服务类业务的增值更多来自于用户基数的增长和用户活动的增加。

  • 发展策略：产品类业务需要不断研发新产品或升级现有产品以保持竞争力，而服务类业务则需要不断优化用户体验和扩大用户群体。

  • 市场适应性：在快速变化的市场和技术环境中，这两类业务都需要灵活适应。产品类业务可能需要更快的迭代周期，而服务类业务则需要不断调整以满足用户的新需求和期望。

产品类业务

对于产品类业务，技术创新是业务发展的主要驱动力。这是因为：

  1. 功能驱动的选择：用户选择产品的主要驱动力是产品的功能能否更好地帮助他们完成任务。功能更强大、性能更先进、体验更顺畅、外观更漂亮的产品更受欢迎。
  2. 技术创新的直接影响：技术创新直接提高产品的功能性和吸引力。例如，苹果通过引入多点触摸操作，UC 浏览器通过开发 U3 内核等，都直接提升了产品的吸引力。
  3. 竞争优势：在产品类业务中，技术创新是获得市场竞争优势的关键。创新不仅提高了产品的功能性，还帮助企业在激烈的市场竞争中脱颖而出。

服务类业务

而在服务类业务中，情况恰好相反，是业务的发展推动技术发展。这是因为：

  1. 规模驱动的选择：在服务类业务中，用户的选择更多基于服务的规模。服务的规模越大，提供的价值和吸引力就越强。
  2. 服务模式创新的重要性：在服务类业务中，服务模式的创新成为业务发展的核心。产品在这里是完成服务的工具或载体。
  3. 技术跟随业务需求：随着服务规模的扩大和业务模式的不断创新，对技术提出了更高的要求。因此，在服务类业务中，技术的发展往往是为了满足业务的需求。

综合分析

  • 互动关系：无论是产品类还是服务类业务，技术与业务之间存在互动关系。在产品类业务中，技术创新通常引领业务发展；而在服务类业务中，业务需求和规模扩张则推动技术进步。
  • 时间轴视角：从更长的时间轴来看，即使是产品类业务，技术的发展也可能由后续业务的扩张和需求变化所驱动。例如，随着 iPhone 功能的增加和普及，其技术也在不断进步以满足新的业务需求。
  • 市场环境的影响：市场环境对这两种类型的业务发展也起着决定性作用。例如，随着市场竞争的加剧，即便是服务类业务，也可能需要通过技术创新来获得竞争优势。

交待任务和询问进度技巧

1. 明确任务目标和期望

  • 具体化任务：当主管交代任务时，确保了解任务的具体目标和预期成果。询问具体的细节和预期的结果，以便你可以更好地规划和执行。
  • 阐述目标：在回应时，重申你对任务目标的理解，确保你和主管对任务的理解是一致的。

2. 识别和聚焦关键点

  • 关键绩效指标（KPIs）：确定哪些指标或成果是衡量任务成功的关键。
  • 关注重点：在讨论进度时，集中讨论这些关键指标的当前状态和任何潜在的挑战。

3. 透明的沟通

  • 真实情况：提供真实、直接的进度更新。如果遇到问题或延迟，请直接说明。
  • 风险和挑战：提前识别并汇报可能影响任务的风险或挑战，以及你的应对计划。

4. 使用简明扼要的语言

  • 避免技术术语：尽量使用非技术性的语言，除非你的主管对相关技术有深入了解。
  • 关键信息优先：首先传达最重要的信息，然后再提供必要的背景或细节。

5. 准备数据和证据

  • 数据支持：使用具体数据和实例来支持你的进度报告和任何观点。
  • 可视化展示：如果可能，使用图表或其他可视化工具来展示复杂的数据或趋势。

6. 提前准备

  • 准备汇报内容：在与主管交流之前，准备一个简明的进度摘要或任务概览。
  • 预期问题：思考主管可能会问哪些问题，并准备好回答。

7. 主动更新

  • 定期更新：不要等到被询问时才提供信息。定期主动更新任务进度，可以建立信任并减少不必要的担忧。
  • 建议和反馈：主动提出建议或寻求反馈，展现你对项目的投入和责任感。

通信和IP修改

概述

连接管理器（CM，Connection Monitor）是一个负责在物理机器之间建立稳定通信的组件。它的主要职责是管理组件之间的通信，确保各组件不需要在代码逻辑中硬编码通信方向和IP地址。这样做的目的是增加整个软件系统的部署灵活性，让系统能适应多变的网络环境和部署场景。

界面展示通道

CM 暴露自己的 API 给外界调用，不加鉴权逻辑

订阅推送通道

定义：CM 与其它组件间建立的长连接进行持续的信息接收响应通道。

谁需要与CM建立：被其他组件依赖的、需要被通知其他组件状态的

规则：

  1. 每个需要与远端建立连接的组件需要与 CM 建立一个订阅推送通道
  2. 订阅推送的动作是一致的
  3. 订阅推送动作为
    • A -> CM 心跳信息
    • CM -> A 心跳回复信息
    • CM -> A 订阅信息更新推送
    • A -> CM 订阅信息更新推送回复

心跳机制

心跳机制设计目标
  1. 差异化心跳：根据组件的重要性和稳定性差异化心跳频率。对于关键组件使用更频繁的心跳，而对于稳定性高的组件则可以降低心跳频率。
  2. 智能心跳：引入机器学习算法，根据过去的数据预测组件的健康状态，从而动态调整心跳策略。
  3. 状态感知心跳：心跳不仅包含是否存活的信息，还可以包括组件的性能指标、资源使用情况等，以提供更全面的健康状态。
  4. 心跳确认机制：不仅发送心跳信号，还需要确认接收方是否收到心跳，以便准确判断组件状态。

心跳数据结构设计目标
  1. 扁平化设计：避免过深的嵌套结构，使数据结构更简单、更易于解析和维护。
  2. 灵活的数据模型：使用如JSON或XML等灵活的数据格式，可以容易地适应不同的需求和变化。
  3. 数据压缩：对心跳数据进行压缩，以减少网络传输的负载。
  4. 版本控制：在心跳数据中包含版本信息，以便在升级或更改心跳数据结构时能够兼容旧版本。
  5. 安全性设计：确保心跳数据的安全性，包括加密传输和验证数据的完整性和真实性。
  6. 故障诊断信息：在心跳数据中包含故障诊断信息，如错误代码和状态描述，以便快速定位和解决问题。

各组件元信息

属性

  • 组件元信息是与各组件源码一起维护的信息集合。
  • 这些信息包括但不限于组件的标识、版本、依赖关系和配置参数。
  • 重要的是，组件元信息不能被CM修改，以保证组件的稳定性和一致性。

支持的操作

  • 连接管理

    • 展示连接状态，包括活跃、非活跃和错误状态。
    • 配置通信方向，支持自动或手动模式。
    • 建立连接，包括自动建立和手动指定。
    • 断开连接，支持自动和手动操作。
    • 刷新连接，可选择自动、手动或强制刷新。

  • 日志展示

    • 显示与心跳异常相关的日志（例如，A->CM）。
    • 记录连接异常事件（例如，A->B）。
    • 跟踪刷新连接的详细信息。

  • 服务注册

    • 实现组件连接配置的注册。
    • 同步组件元信息。

  • 服务订阅发布

    • 根据组件订阅的信息建立长链接并持续发送心跳（例如，A -> CM）。
    • 在B、C、D等组件发生异常等场景时，向A推送通知。

  • 依赖关系管理

    • 组件层面

      • A -> B 接口版本依赖
      • A -> CM 接口版本依赖
      • SC -> CM 接口版本依赖

    • 属性层面

      • 强依赖

        • 无法连接，不支持组件间的通信
        • 总控界面报错

      • 弱依赖

        • 界面部分功能受到影响，但不会产生恶劣影响

场景

心跳消失

  • 思路：将心跳消失视为组件不健康的信号，并同步更新状态。
  • 操作：在状态管理中更新相应的组件状态为异常。

心跳元信息不同步

  • 目的：确保所有组件的心跳元信息保持同步。
  • 操作：进行必要的检查和调整，以确保所有组件的心跳信息是最新的。

通信异常

  • 处理策略：将通信异常视为潜在的攻击或系统故障。
  • 操作：采取相应的安全措施，如隔离受影响的组件，记录详细日志以供分析。

额外注意事项

  • 安全性：CM应该具备防御网络攻击的能力，包括但不限于拒绝服务攻击（DoS）和分布式拒绝服务攻击（DDoS）。
  • 可扩展性：设计CM时需要考虑到系统的可扩展性，确保在组件数量增加时仍能高效运行。
  • 用户界面：提供一个直观的用户界面以便管理员能够轻松管理连接和监视系统状态。

展望

展望

  1. 未来发展：描述CM如何适应未来技术变化和扩展需求。
  2. 技术升级计划：预计的技术迭代和更新计划，以及对现有系统的影响评估。
  3. 集成其他系统：计划与哪些系统集成，以及集成的潜在好处和挑战。
  4. 性能优化：针对性能瓶颈的分析和优化策略。

安全措施

  1. 身份验证和授权：确保只有授权用户/系统可以访问CM的API。
  2. 数据加密：在传输过程中对数据进行加密，保护敏感信息。
  3. 安全审计：记录和监控所有对CM的访问和更改。

性能指标

  1. 响应时间：定义CM响应请求的时间标准。
  2. 吞吐量：CM可以处理的最大请求量。
  3. 可用性：CM的预期运行时间和维护时间。

故障恢复策略
